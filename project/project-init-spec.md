# `/project-init` — Spec

> Scaffolds a project substrate (the structured layout from
> `project-substrate-notes.md`) in the current repo. First skill in
> the substrate suite. Does not migrate legacy content — that's
> `/project-migrate`'s job.

## Purpose

Create the directory layout, conventions, and ambient instructions
that let the substrate suite (`/project-lint`, `/project-sync`,
`/project-promote`, `/project-migrate`) operate. Makes the substrate
*real* — files on disk that foreground Claude and imp primitives can
read and write.

## Scope

**Does:**
- Creates `project/` (or configured location) with the standard
  layout, kind directories, and stub README files explaining each.
- Writes `project/_meta/conventions.md` (the canonical spec for
  frontmatter, kinds, drift semantics).
- Writes a starter `project/log.md`.
- Adds a substrate-aware block to repo `CLAUDE.md` (creates if
  missing) so foreground Claude knows how to read/write.
- Adds `<repo>.project-proposals/` to `.gitignore` if not already
  ignored. (imp's private scratch lives elsewhere — see H8 in
  substrate notes — and is not the user's concern at init time.)

**Does not:**
- Read or migrate legacy docs — `/project-migrate` is separate.
- Generate concept pages — those are auto-generated by
  `/project-sync` once content exists.
- Set up scheduled routines — that's the user's call via `/schedule`
  after init lands.
- Modify any code outside `project/`, `.gitignore`, and `CLAUDE.md`.

## Invocation

```
/project-init
/project-init <path>     # location for the substrate; default project/
/project-init --force    # overwrite existing stub READMEs (never user content)
```

Interactive prompts on first run:
1. Confirm substrate location (default `project/`, or whatever the
   user typed).
2. If location already exists with content, refuse and direct to
   `/project-migrate`.
3. Confirm CLAUDE.md edit — show diff, get sign-off.
4. Confirm `.gitignore` edit — show diff, get sign-off.

After completion: print a short "what's here, what's next" summary
and a pointer to `/project-migrate` for brownfield projects.

## Layout produced

```
project/
├── README.md                  # explains the substrate to a reader
├── log.md                     # append-only chronological
├── _meta/
│   └── conventions.md         # frontmatter spec, kind definitions, drift semantics
├── rules/
│   └── README.md              # "rules go here. Drift = alarm. Examples..."
├── aspirations/
│   └── README.md              # "aspirations go here. Contradictions are OK..."
├── learnings/
│   └── README.md              # "learnings go here. They age in relevance..."
├── plans/
│   ├── active/
│   │   └── README.md
│   └── archive/
│       └── README.md
├── tasks/
│   └── README.md              # or root TODO.md, configurable
├── concepts/
│   ├── README.md              # "AUTO-GENERATED. Do not edit by hand."
│   └── .gitkeep
└── reference/
    └── README.md
```

Each kind's `README.md` includes:
- One-paragraph definition of the kind (lifted from H0 in the
  substrate notes — drift semantics, source of truth, what belongs
  here, what doesn't).
- A minimal example file with frontmatter.
- A pointer to `_meta/conventions.md` for the full spec.

## Frontmatter conventions

Written to `project/_meta/conventions.md`. Every entry in the
substrate has frontmatter:

```yaml
---
kind: rule | aspiration | learning | plan | task | reference
title: short human-readable title
created: 2026-05-09
updated: 2026-05-09
provenance:
  source: human | imp-build:R-NNN | imp-research:R-NNN | imp-wiki:R-NNN | migration:M-NNN
  promoted_at: 2026-05-09       # only if promoted from a candidate
  promoted_by: human | claude
---
```

Per-kind extensions:

| Kind | Extra fields |
|---|---|
| `rule` | `enforces` (optional list of file globs the rule applies to) |
| `aspiration` | (none) |
| `learning` | `relevance_horizon` (optional date past which to fade), `topics` (list) |
| `plan` | `state: exploring \| active \| shipped \| shelved \| abandoned`, `supersedes` (list of plan filenames), `companion_dir` (optional bool — true when `<slug>/` companion dir exists alongside) |
| `task` | `topic` (optional), `parent_plan` (optional) |
| `reference` | `subject` (what external thing this references) |

The `concepts/` namespace is auto-generated and uses different
frontmatter (sources list — see substrate notes H2). Init writes a
note to `concepts/README.md` explaining this.

## CLAUDE.md snippet

Init appends (or creates) a section in the repo's root `CLAUDE.md`:

```markdown
## Project substrate

This repo uses a structured project-knowledge substrate at `project/`.
Read from it before answering questions about design, intent, or current
behavior. You may edit substrate files directly when capturing decisions,
learnings, or plans. imp may not — it produces proposals you (or the
human) review and approve.

### What's where

- **`project/concepts/<topic>.md`** — auto-generated synthesis pages.
  Start here for a topic overview. Sources are listed in frontmatter.
  Don't hand-edit; regenerated by `/project-sync`.
- **`project/rules/`** — locked-in constraints. Code violating these is
  a bug. Internal contradictions are alarming.
- **`project/aspirations/`** — what we're going for. Contradictions are
  fine; code falls short by definition.
- **`project/learnings/`** — discovered knowledge that informs caution
  but isn't a rule. Decays in relevance, not in truth.
- **`project/plans/active/`** — what's being worked on now.
- **`project/plans/archive/`** — historical and shelved plans. Useful
  for "yeah but why?" but not authoritative on current behavior.
- **`project/tasks/`** — task tracking.
- **`project/log.md`** — append-only chronological history.

For drift semantics per kind, see `project/_meta/conventions.md`.

### imp proposals

imp produces proposals at `<repo>.project-proposals/P-NNN-<slug>.md`
when scheduled sweeps detect promotion candidates, drift, doc rot, or
similar. Review pending proposals via `/project-promote` (or skim the
directory directly). Approval gradient:

- **Always-safe** (auto-apply): `log.md` appends, archive moves.
- **Claude-approvable**: new learning entries, concept regeneration,
  candidate flags. You can approve these on the user's behalf.
- **Human-required**: rules edits, deletions, supersede markers,
  anything that loses information. Surface to the user; don't auto-apply.
```

The exact wording is open — the snippet should be short enough that
the user is unlikely to remove it for noise reasons, and specific
enough that Claude's behavior changes when it reads it.

## Idempotency and safety

- Running on an existing substrate: detect via presence of
  `project/_meta/conventions.md`. Default behavior: no-op + status
  report. With `--force`: re-write only the stub README files and
  `_meta/conventions.md` (these are skill-owned). Never touch
  user-authored content.
- Running where `project/` exists with non-substrate content (e.g.
  dreamlands' current state): refuse, direct to `/project-migrate`
  with a hint about `--into project-new/` if the user wants to keep
  the legacy dir untouched.
- All file writes are idempotent and resumable — if the skill is
  interrupted, re-running completes the rest cleanly.

## Configuration

A single optional file at `project/_meta/config.yaml`:

```yaml
location: project/                       # where the substrate lives (relative to repo root)
proposals: ../<repo>.project-proposals/  # where imp proposals land (sidecar, gitignored)
tasks_path: project/tasks/               # or TODO.md if user wants flat task tracking
```

Defaults are sensible; the file is created with all keys commented
out so users see what's available. No mandatory configuration — the
defaults work for greenfield init.

Note: imp's *private* work area (research scratch, build worktrees)
is sized and located by imp itself per the existing `<repo>.researches/`
and `<repo>.worktrees/` patterns. Init doesn't configure that.

## Open decisions (during build)

- **Tasks: directory or root TODO.md?** dreamlands keeps a flat
  `TODO.md` at root. Some users will want that; others will want
  per-topic task files under `tasks/`. Configurable via
  `tasks_path`; default to `project/tasks/` and let users opt out.
- **Seed `aspirations/<project>.md` from README?** Some projects
  have a README that's mostly aspiration-shaped. Optional interactive
  step: "I see a README. Want me to seed an aspirations entry from
  it?" — default no.
- **`concepts/_topics.yaml` for topic seeding?** Q2 in substrate
  notes. For init, leave it absent — let `/project-sync` create it
  the first time it runs and discovers topics.
- **Edit `CLAUDE.md` or create a sibling `CLAUDE.project.md`?** Lean
  toward editing `CLAUDE.md` directly with a clearly-fenced section,
  to avoid file proliferation. Open to redirect.

(Decisions that were open and have closed: staging dir creation —
init does not create it; the proposals dir is the only sidecar
init-time gitignore covers, and it's lazy-created on first proposal.)

## Build order

1. Implement the layout writer (pure file creation, no model calls).
2. Implement the `_meta/conventions.md` content (writes the
   frontmatter spec and kind definitions verbatim from substrate
   notes H0).
3. Implement the per-kind README content (lifted from H0, one
   paragraph each).
4. Implement the CLAUDE.md / .gitignore edits with diff-and-confirm.
5. Implement the existing-content detection and refusal path.
6. Implement `--force` re-write of skill-owned files only.
7. Manual test on a fresh repo.
8. Manual test on imp's own repo (greenfield-ish — current `project/`
   is plans-only, no substrate structure yet). Use `--into
   project-substrate/` if needed to avoid clobbering existing plans.

## Done when

- A fresh repo gets a working substrate scaffold.
- imp's own repo gets a substrate scaffold (probably under a
  separate dir to avoid disrupting current plan docs).
- Foreground Claude, after reading the new CLAUDE.md section,
  correctly orients to substrate paths when asked about project
  knowledge.
- The next skill in the chain (`/project-migrate` or `/project-sync`,
  whichever lands next) finds the layout it expects.
