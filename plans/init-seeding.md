# Substrate seeding at init time

## Problem

`imp init` (Substrate/ProjectInit.cs) scaffolds an empty substrate:
template files copied, dirs created with `.gitkeep`, CLAUDE.md
upserted. On a brand-new repo that's the right amount of work. On a
non-empty repo with existing intent, conventions, and history, the
substrate starts in a state where the gnome has nothing to anchor on
and the human reader sees an empty directory tree that says nothing
about their project.

There's a window right at init time where the parent coding agent
(Claude in the conversation that ran `imp init`) has rich context the
substrate doesn't yet capture: what the user just told it, what's in
the existing CLAUDE.md / README, why this codebase exists. We should
use it.

## Approach: parent seeds small; everything else accretes

After `imp init` finishes, the parent agent makes a single seeding
pass. The pass writes only what the parent already knows from
conversation context — **no source-reading-for-comprehension**, no
per-file enumeration, no exhaustive coverage. The output should look
like 3–8 short, high-signal files, not a finished substrate.

The forcing test for what gets written: *can the parent author this
without re-reading the code?* If yes, terse parent-written entry. If
no, defer — either to qwen later (per-file digests on demand) or to
natural accretion through `imp note` + `imp tidy` during real work.

Rejected alternatives:

- **Parent writes everything.** Overfits to one moment of
  conversation context. Burns parent budget on reading-comprehension
  tasks the parent isn't faster at than qwen. Doesn't exercise the
  research path that maintains substrate ongoing.
- **Skeleton + queued qwen tasks for full coverage.** Front-loads
  research budget on speculative coverage that may never get read.
  qwen research quality on cold-read tasks is uneven (memory:
  qwen3-coder hallucinates API shape on this codebase). Mismatches
  the substrate's natural shape, which is accretive, not bulk-loaded.

## What the parent writes

Kind-by-kind, with the test "do I know this from conversation, not
from reading code":

- **`imp/learnings/`** — yes. Why-decisions and gotchas the user has
  stated or that are explicit in CLAUDE.md/README. One file per
  distinct learning, terse (often 5–15 lines body). Examples on this
  repo: "rewritten from MCP server to CLI to escape stderr opacity"
  (in CLAUDE.md), "qwen3-coder hallucinates API shape, route to
  research-only" (a stated fact), "no local FS paths in code"
  (rule-adjacent, but the *why* is a learning).
- **`imp/reference/`** — yes when URLs are referenced in
  conversation, README, or CLAUDE.md. One per source, with the
  local snippet captured. Don't speculatively crawl.
- **`rules/`** — yes for hard invariants the user has explicitly
  stated as invariants (not preferences, not "we usually"). On this
  repo: "imp owns the auto-commit; model must not run git commit"
  is rule-shaped.
- **`plans/`** — yes for in-flight work the user has described and
  not yet shipped. State `exploring` if the shape isn't decided,
  `active` if work is ongoing.
- **`TODO.md`** — yes for anything the user has mentioned wanting
  done that isn't already a plan.

## What the parent does NOT write

- **`imp/concepts/`** — gnome territory. Auto-generated narrative
  syntheses, regenerated by `imp tidy`. Hand-writing here gets
  overwritten.
- **`imp/_index/`** — gnome territory. Per-file/symbol/feature
  digests need code reads at the granularity the parent should not
  pay for. Defer.
- **`imp/note/`** — capture surface only. `imp note` writes here;
  the gnome processes it. The parent shouldn't write notes on the
  user's behalf.
- **`imp/log.md`** — append-only, gnome-maintained.

The parent also doesn't try to be exhaustive across the kinds it
*does* write. Three good learnings beat ten thin ones.

## Follow-on: `/project-migrate` for legacy content

After the seed pass, the init-seed skill should check for legacy
content the seed deliberately didn't touch (`project/` dir with
non-substrate content, or any pre-existing design docs at the repo
root beyond README/CLAUDE/TODO). If found, the skill ends by
suggesting `/project-migrate` — that's the path for ingesting
existing prose into structured substrate entries, distinct from the
parent-knowledge seed this skill handles.

Sequence: `imp init` → init-seed skill (this plan) → if legacy
detected, `/project-migrate` (Phase 2 plan). The seed and migrate
flows are complementary: seed captures what the parent knows from
conversation; migrate captures what existing docs preserve. Neither
subsumes the other, and running both is the normal init UX for a
non-empty repo with prose history.

## Filling the gaps later

The substrate accretes through normal use. The seed plus a few
weeks of `imp note` captures plus regular `imp tidy` runs gets the
substrate to a useful state without one big upfront read pass.

For the specific case of `imp/_index/by-file/<path>.md` (the cold
read that's most useful and most expensive), defer until either:

1. A `note` references the file, so `imp tidy` has a real reason
   to look at it, or
2. The user explicitly asks (`imp index <path>` or equivalent — not
   designed yet).

Bulk-filling `_index/` at init time is the option this plan rejects,
but it remains available as a future `imp index --all` if the
deferred-filling path proves too slow in practice.

## How the parent knows to do this

`imp init`'s closing output already prints a "Next:" section. Add a
seeding nudge there pointing at a skill (or this plan) that walks
the parent through the kind-by-kind decisions above. Two
implementation options:

- **Option α: extend the existing `imp` skill.** The parent already
  loads `skills/imp.md` for build/research/tidy guidance. Add a
  "post-init seeding" section. Pro: one skill, no new file. Con:
  init-time guidance is read at init time and never again, but the
  skill is loaded every imp invocation.
- **Option β: dedicated `init-seed` skill.** New file, loaded only
  when the user runs init. Pro: scoped. Con: skill discovery is
  best-effort; relies on init's "Next:" output telling the parent
  to load it.

Lean β. The seeding pass is a one-time event; bundling it into the
always-loaded skill bloats the always-on prompt for no benefit on
the 99% of imp invocations that aren't init.

## Open questions

- How much of "the parent knows from conversation context" should
  be auto-captured by `imp init` itself? Right now CLAUDE.md is
  read at session start; init could grep it for stated
  invariants/learnings and pre-populate stub files. Probably not —
  parsing prose for substrate kinds is exactly the brittle thing
  we're trying to avoid. Better to let the parent (which already
  has the prose loaded) do the structuring.
- Should the seeding pass run in the same conversation as `imp
  init`, or be a separate user-triggered step (`imp init --seed`,
  or just "run /init-seed afterward")? Same conversation is cheaper
  context-wise; separate is more controllable. Default to same
  unless we hit a reason not to.
- What's the qwen-fallback threshold for `_index/by-file/` — fully
  on-demand, or batched per-tidy-run when notes pile up referencing
  files without index entries? Defer to first dogfood.
